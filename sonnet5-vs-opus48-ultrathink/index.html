A Kriya Lab · max effort · 01 July 2026

We turned the
dial to max.

Same models. Same real Apolaki build. Same three Kriya tasks. This time both Sonnet 5 and Opus 4.8 ran at maximum reasoning effort — an identical ultrathink directive on both arms.

The question isn't who wins. It's: what does more thinking actually buy — and what refuses to move?

The question

At default effort, they were close.
Is “max” worth paying for?

Cranking effort roughly doubles the thinking. Three things could happen: it finds more bugs, it changes a verdict, or it just runs up a bigger bill.

We measured all three. Two of them surprised us.

Baseline: the default-effort lab. Everything here is the same suite re-run with a matched max-rigor directive.

First: did “max” actually do anything?

Yes — 1.5–2.2× more thinking.

Thinking tokens bill as output. Both models generated far more of them than at default effort — proof the directive changed behaviour, not just the label.

Opus · default18,913
Opus · max35,667
Sonnet · default29,414
Sonnet · max52,170

Total output tokens across the three tasks. Per task the jump ran 1.54×–2.21× (Opus's design answer alone grew 2.2×).

What max effort bought · recall

More thinking → more findings. Really.

On the two objective tasks, both models genuinely found more valid defects at max effort — not noise, real issues they'd missed.

TaskSonnet 5 · default → maxOpus 4.8 · default → max
Gotcha catch — valid defects6 → 78 → 9
Adversarial gate — findings9 → 1311 → 14
Seeded defects still caught3 / 33 / 3

The new catches were real: Opus flagged that the two guards can be mounted out of order; Sonnet spotted an error-code enumeration oracle. Recall rose — for ~1.7× the cost (next-but-one slide).

Surprise #1 · what it did NOT change

The verdict never moved.

Every gate decision was identical at default and max effort. Both models vetoed the fail-open gate at both settings; both flagged the diff as needs-fix at both settings.

DecisionSonnet 5Opus 4.8
Gate task — defaultVETO ✓VETO ✓
Gate task — maxVETO ✓VETO ✓
Gotcha task — both effortsNEEDS-FIX ✓NEEDS-FIX ✓

Max effort bought a longer list, not a different answer.

Surprise #2 · the blind spot that survived

Effort didn't fix the blind spot.

At default effort, Sonnet 5 uniquely caught a subtle bug — a stateful g/y regex flag that makes authorization nondeterministic. We hoped max-effort Opus would find it too.

Sonnet 5 at max
  • Converged upward — picked up Opus's default strengths it had missed (unknown-role-should-deny, permission, boundary checks).
  • Kept its signature stateful-regex catch.
Opus 4.8 at max
  • Found 3 more issues (HEAD→GET, trust invariants, dead rules)…
  • …but STILL missed the stateful-regex bug — even with 2× the thinking.

Some blind spots are model-characteristic, not effort-limited. You can't out-think them — you have to ensemble around them.

The engineering takeaway

Max effort is a scalpel, not a default.

  • Pay for max on enumeration, not on decisions. Verdicts were effort-robust — spend the extra thinking only when you need the full defect list, not a PASS/VETO.
  • Ensemble for the highest stakes. Blind spots didn't close with effort. For critical audits, union Opus + Sonnet rather than trusting one at max.
  • Cap Sonnet 5's output. Its cheaper tokens only pay off once you rein in the 1.5–1.9× verbosity.
  • Front-load Sonnet work before 31 Aug 2026 while intro pricing makes it ~40% cheaper.

These fold straight into the routing recommendation that Kriya consumes — which is the whole point of running these.

The honest fine print

What this is — and isn't.

  • “Max” = an identical ultrathink directive on both arms (headless mode has no literal effort knob). Effort is held equal, and the 1.5–2.2× token jump confirms it landed.
  • One run per task. Signal, not a benchmark.
  • Graded against published answer keys by the author, not a blind panel — objective tasks have ground-truth defects.
  • Served model verified from billing telemetry for all 6 runs; generated code discarded; costs at official published rates.
The takeaway

Point max effort at
enumeration, not decisions.

More thinking found more bugs, never changed a verdict, and couldn't fix a model's blind spot. So spend it deliberately — and ensemble when it matters most.
