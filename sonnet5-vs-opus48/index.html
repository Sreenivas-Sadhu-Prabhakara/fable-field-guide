The test bed · Kriya

Why grade models inside Kriya?

Kriya is an agentic SDLC orchestrator that walks a project from intake to production through fixed, spec-driven stages — with adversarial roles that make it a brutal test of a model's judgement.

The four pillars

  • Producer ≠ critic. The agent that writes is never the one that approves.
  • Shift-left NFRs. Security, performance & edge cases are specified up front, not bolted on.
  • Traceability spine. Every requirement gets an ID: BR → FR → component → test → deploy-check.
  • Tiered gates. Humans decide the irreversible; agents handle the rest.

What that demands of a model

  • Design rigor — write a complete, traceable spec with threats & budgets.
  • Bug recall — catch real defects in brownfield code, not toy bugs.
  • Adversarial spine — refuse to rubber-stamp; veto a bad gate.

Kriya's default model is already claude-opus-4-8. The question is whether Sonnet 5 can share the load.

The lab · method

One build. Three Kriya-shaped tasks.
Identical inputs. Both models.

TASK 1 · DESIGN

Shift-left LLD

Produce a full LLD slice for Apolaki's bill-photo onboarding: BR/FR IDs, STRIDE threat model, edge catalog, NFR budgets, traceability.

TASK 2 · BUILD

Brownfield gotcha catch

Review a real diff against Apolaki's live auth model. Do they catch the seeded defects — installer≠dealer, owner-scoped consent, elevated-role bypass?

TASK 3 · GATE

Adversarial verifier

A central authorize() with a planted fail-open bug. Act as Kriya's independent gate: find it, rank it, PASS or VETO.

Anchored on real Apolaki source (its policy matrix, role-normalization, consent model). Both models run at matched effort with byte-identical prompts. Every run captured Claude Code's authoritative billing telemetry — the served model, tokens, and cost are measured, not claimed.

Result · Task 3 · adversarial gate

Both vetoed the fail-open.
Each caught a bug the other missed.

The planted defect — an unmatched path returning { allow: true } — was caught by both, both correctly output GATE: VETO. Neither rubber-stamped.

Sonnet 5 · 9 findings · uniquely caught
  • Stateful g/y regex flag → nondeterministic authz across requests. The subtlest bug in the whole lab.
  • Uncaught exception in the authorizer fails open in practice
Opus 4.8 · 11 findings · uniquely caught
  • Case-sensitive method compare → drops to the fail-open default
  • Policy-author vs runtime role-normalization mismatch
  • ReDoS surface on attacker-controlled path

Call it a tie on rigor: Opus enumerates more footguns; Sonnet 5 reaches one level deeper into runtime behavior.

Result · Task 1 · shift-left design

Two great specs —
with different instincts.

Both delivered the full LLD: complete BR→FR→test traceability, STRIDE threat model, numeric NFR budgets — and both independently reinvented Apolaki's real storage-unconfigured fallback (a genuine characterization test in the codebase).

Opus 4.8 · systems rigor
  • Tighter numeric budgets — upload p95 ≤ 800ms, OCR ≤ 8s, delete propagation ≤ 60s
  • Data minimization: discard extracted PII post-OCR; store only kWh + tariff
  • Every test wired to an explicit deploy-gate
Sonnet 5 · domain awareness
  • PH-specific edges: co-op & prepaid meters, Tagalog/English OCR
  • Fraud-via-edit, OCR-vendor DPA, backup-retention, implausible-value range checks
  • Threat model reads like someone who's seen a real Philippine electricity bill

Neither is “better.” Opus engineers the system harder; Sonnet 5 understands the world around it better.

The verdict

No blowout. A division of labour.

Opus 4.8 wins when…
  • the task is the final gate — it enumerated the most footguns, missed the fewest
  • you need tight, minimal, exhaustive output — more findings in fewer tokens
  • engineering rigor & budget discipline matter more than sticker price
Sonnet 5 wins when…
  • you want breadth & domain awareness — it saw the real-world edges
  • it's a high-volume producer role and intro pricing is live
  • occasionally it reaches deeper — the stateful-regex catch Opus missed

Sonnet 5 is genuinely close to Opus on this pipeline. It does not dominate — and it isn't the automatic “switch and save” the token price implies.

What it means for Kriya

Don't pick one model.
Assign roles.

Kriya already runs data-driven per-agent model + effort tiers. The evidence points to a mixed roster:

  • Sonnet 5 as the producer swarm — design drafts, breadth, domain modeling — especially while intro pricing holds.
  • Opus 4.8 as the veto gate & critic — the role where exhaustive enumeration and the fewest misses matter most.
  • Tame Sonnet's verbosity with lower effort tiers / output caps — that's where its cheaper tokens finally pay off.

Exactly the call Kriya's tech-radar exists to make — now with real numbers behind it.

What this is — and isn't

The honest fine print.

  • One run per task. Signal, not a benchmark — no averaging over seeds.
  • Graded against published keys by the author, not a blind expert panel.
  • Effort was held equal, not pinned to a literal “max.” Headless mode doesn't expose a max-effort knob; both ran at Claude Code's identical default.
  • Kriya's method, not its binary. No API key in the lab environment, so tasks ran via Claude Code headless — Kriya's pillars applied by hand, not the Go orchestrator.
  • Sonnet 5 was < 24h old. Its behaviour and routing are still settling.
  • Cost normalized to official published rates, not Claude Code's internal table.

Everything here is reproducible: identical prompts, headless CLI calls, and billing-telemetry verification. The models' generated code was discarded, as agreed — only the measurements remain.

The takeaway

The premium is real,
but it's now a choice, not a default.

Sonnet 5 earns a seat at the table. Opus 4.8 still earns the gate. The winning move is to run both — and let Kriya decide which agent gets which.
